Duda frecuente

¿interface o type?

// Ambos parecen similares
interface User {
  name: string;
  age: number;
}

type User = {
  name: string;
  age: number;
}

// ¿Cuál es la diferencia? ¿Cuándo usar cada uno?

Ambos definen formas de objetos, pero tienen superpoderes distintos.

interface: extensible, declaración de merging
type: uniones, intersecciones, primitivos
Diferencias clave

Interface vs Type

// ✅ Interface: extensible (declaration merging)
interface User { name: string }
interface User { age: number } // se fusiona
const user: User = { name: "Ana", age: 30 };

// ✅ Type: uniones y tuplas
type ID = string | number;
type Status = "active" | "inactive";
type Tuple = [string, number];

// ❌ Type no se puede reabrir
// ❌ Interface no soporta uniones primitivas

Cada uno tiene su terreno.

Regla de oro: interface para objetos, type para todo lo demás
95% de los casos
Regla definitiva

Usa interface para objetos, type para el resto

Interface → formas de objetos (props, state, API responses)

Type → uniones, intersecciones, tuplas, tipos primitivos, utilidades

// ✅ Bueno
interface Product {
  id: number;
  name: string;
}
type ProductID = Product["id"]; // number
type Status = "draft" | "published";

// ⚠️ Excepción: si necesitas union types o mapped types, usa type

Sigue esta regla y tu código será consistente y predecible.

Conclusión

Adiós a las dudas

Interface → objetos que pueden extenderse (declaration merging)

Type → uniones, tuplas, primitivos, utilidades avanzadas

Ambos son compatibles, pero esta regla mantiene tu código limpio

Regla simple:
¿Es un objeto que podría crecer? → interface
¿Es una unión, tupla o primitivo? → type
